## Things that went sideways

### Bottle wipe (mid-morning)
Tried to swap only `wine` + `wineserver` + `winevulkan.dll` into the engine without the matching `lib/wine/` library tree. New wine binary didn't recognize the existing prefix layout (version mismatch with the old library trees), treated it as uninitialized, and **re-initialized the bottle** — wiped the SWG game client + launcher + per-program plist. No Time Machine snapshot, no APFS local snapshot. Recovery: deleted the bottle, fresh install. The fix in the afternoon was to do `make install DESTDIR=staging` and bulk-replace the whole `lib/wine/x86_64-*` trees atomically alongside the binaries. **Lesson: never partial-swap a Wine engine.**

### Mac became unresponsive
Caused by ~965 stuck `winedbg --auto` processes accumulating over hours of SWG launcher JIT crashes + one runaway `conhost.exe` with 489 min CPU. Load avg hit 75. **Fix in `66713d2`:** `AeDebug=0` on new bottles stops Wine from auto-spawning the debugger. Documented in `~/.claude/.../memory/wine-aedebug-storm.md`.

### Locale plist decode bug (last hour)
Manually wrote a per-program plist with `<string>auto</string>` for locale. `Locales.auto.rawValue` is the empty string `""`, so `PropertyListDecoder` threw `dataCorrupted`. My `loadProgramSettings` caught + returned nil → no env vars merged → sliver-render kept happening. Fixed the plist (now uses `<string></string>`). **Follow-up to ship:** make `loadProgramSettings` salvage a partial decode — read just the `environment` dict if full decode fails. Then manually-edited plists won't silently no-op.

## What's not done

- **`loadProgramSettings` salvage path** for partial plist decodes (noted above).
- **Install-flow auto-plist for Java apps.** When `Wine.runProgram` is called from the install flow against a self-contained Java exe (like SWG launcher), auto-write a default plist with `_JAVA_OPTIONS=-Dprism.order=j2d -Xint` so first-launch doesn't sliver-render.
- **Post-install scan false-positive on portable apps.** `userVisiblePrograms.isEmpty` warning fires for self-contained .exe apps that don't drop files in Program Files. Detection should recognize "the .exe you picked is the program."
- **Run button grayed-out logic.** Crosswire's program scan only finds .exes inside `<bottle>/drive_c/Program Files*/`. Portable apps launched from elsewhere never become "programs." Either copy-into-bottle on install (done manually today) or extend the program scan to track user-picked .exes outside drive_c.
- **Auth-wedge in SWG launcher** (CLOSE_WAIT on auth socket after Login click). Pre-dates today's work; deep Wine SSL/winhttp bug. Symptoms last seen this morning; not retested today.
